## Migration Strategy (When Needed)

### 1. Adding new columns
```sql
ALTER TABLE <table_name> ADD COLUMN <column_name> <type> DEFAULT <value>;
```
SQLite supports ADD COLUMN but not DROP COLUMN (pre-3.35) or ALTER COLUMN.

### 2. Adding new tables
Already handled by `init_db()` which calls `Base.metadata.create_all()`.
New models are automatically created on startup (CREATE IF NOT EXISTS behavior).

### 3. Changing column types
1. Create new column with desired type
2. Copy data: `UPDATE <table> SET new_col = CAST(old_col AS <new_type>)`
3. Drop old column (SQLite 3.35+) or recreate table

### 4. Renaming columns
SQLite 3.25+ supports `ALTER TABLE RENAME COLUMN`. For older versions:
1. Create new table with correct schema
2. Copy data from old table
3. Drop old table
4. Rename new table

### 5. For production deployment
Set up Alembic when Bob has multiple users or needs zero-downtime migrations:
```bash
pip install alembic
alembic init alembic
# Configure alembic.ini with database URL
# Generate migration: alembic revision --autogenerate -m "description"
# Apply: alembic upgrade head
```

## Known Schema Issues
- `skill_runs` table had AUTOINCREMENT issues on SQLite (fixed by Bob Doctor service)
- Two databases existed historically (`.mind-clone/` and `AppData/Local/`)
- Now consolidated to single `.mind-clone/mind_clone.db`
- `SchemaMigration` table exists for tracking but no formal migration framework uses it yet

## Schema Tracking
The `schema_migrations` table stores applied migration versions with checksums.
Currently used for manual tracking only -- not connected to an automated tool like Alembic.
